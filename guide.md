# SEN3006 - Software Architecture

# Java Design Pattern Project

## 1. Project Description

In this project, students are required to first define a software problem clearly. Based on
this problem definition, they must analyze and choose at least two appropriate Design
Patterns that best address the problem. They will then implement the solution in Java and
document their work in a detailed report. This project aims to enhance students'
understanding of software design principles and best practices.

After submission, students will be required to present their project in an in-person session,
where they will explain their problem definition, justify their choice of design patterns, and
demonstrate their implementation.

## 2. Constraints and Rules

### 2.1 Design Pattern Selection

Students must choose at least two of the following design patterns:

- **Creational Patterns**: Singleton, Factory Method, Abstract Factory
- **Structural Patterns**: Adapter, Decorator, Facade, , Composite
- **Behavioral Patterns**: Observer, Strategy, Command

### 2.2 Coding Requirements

The project must be implemented in Java (Java 8 or later) and adhere to SOLID
principles. It should include at least three classes (including interfaces and abstract
classes) and must be executable via a `Main.java` class. During the presentation, students
are expected to demonstrate test cases in the `Main.java` class.

### 2.3 Use of AI Tools

The use of AI-assisted tools (such as ChatGPT, GitHub Copilot, or similar tools) is
permitted and may even be beneficial for tasks such as understanding concepts,
generating example code, or improving documentation.

However, students are expected to fully understand all parts of their project, including the
design decisions, UML diagrams, and implementation details. AI tools should be used as
supportive tools, not as a replacement for learning and understanding.

During the project presentation and question–answer session, students are responsible for
clearly explaining:

- the problem definition,
- the selected design patterns,
- the system architecture,
- and the Java implementation.

Students must be able to justify and explain every part of their project. If a group is
unable to explain how their system works, it may negatively affect the evaluation.

## 3. Report Requirements

It is strongly recommended that you structure your report according to the TÜBİTAK
2209 project report format. The English version of the report template is currently not
available on the official web page; however, it will be shared with you. The report is
expected to be approximately 10–14 pages, excluding code listings and appendices.

Your report should be structured using the following sections.

### 3.1 Introduction

Provide a clear introduction to the problem addressed in the project.
This section should include:

- Background information about the problem
- Motivation for the project
- Objectives of the system to be developed
- A brief overview of the proposed solution

### 3.2 Problem Definition and System Requirements

Clearly define the software problem you are solving.
This section should include:

- Description of the problem scenario
- System requirements
- Functional requirements
- Non-functional requirements (if applicable)

Explain why a structured software architecture approach is needed for this problem.

### 3.3 Design Pattern Explanation

Explain the selected design patterns in detail.
For each design pattern include:

- Definition of the design pattern
- When and why it is used
- Advantages of using the pattern
- Why this pattern is suitable for your project
- A real-world example where the pattern is applied

Students must explain at least two design patterns used in the project.

### 3.4 System Design and UML Diagrams (Mandatory)

The following UML diagrams must be included in the report with detailed
explanations.

#### Mandatory Diagrams

- **Class Diagram**
  Illustrates the structure of classes, interfaces, inheritance relationships, and associations.
- **Sequence Diagram**
  Shows object interactions and method calls during system execution.
- **Use Case Diagram**
  Describes system functionality and user interactions.

#### Additional Diagrams (Optional but Recommended)

Depending on the complexity of your project, you may also include:

- **Activity Diagram**
  Useful for representing workflows and decision logic.
- **State Diagram**
  Recommended if your system includes different object states.
- **Component Diagram**
  Useful for larger systems with modular architecture.
- **Deployment Diagram**
  Applicable for distributed or multi-layer systems.

Students should decide whether additional diagrams are needed to explain their system
effectively.

### 3.5 Implementation (Code Explanation)

Explain how the UML design is translated into Java code.
This section should include:

- Explanation of the main classes
- Explanation of interfaces and abstract classes
- How the selected design patterns are implemented
- Explanation of key methods
- Code structure and relationships between classes

Screenshots or code snippets may be included if necessary.

### 3.6 Testing and Demonstration

Describe how the system was tested.
Include:

- Test scenarios
- Example inputs and outputs
- Demonstration of the system execution
- Explanation of test cases implemented in Main.java

### 3.7 Results and Evaluation

Evaluate the effectiveness of your implementation.
Discuss:

- Advantages of the selected design patterns
- Limitations of your implementation
- Possible improvements to the system
- Alternative design patterns that could be used

### 3.8 Conclusion

Summarize the project and its contributions.
Include:

- What was achieved in the project
- Key lessons learned
- Importance of design patterns in software architecture

### 3.9 References

List all references used in the project. These may include:

- Books
- Academic papers
- Documentation
- Online resources

All sources must be properly cited.

## 6. Project Presentation

After the submission deadline, each student is required to present their project in an in-person session. The presentation should cover:

- Explanation of the selected design patterns and why they were chosen.
- UML diagrams and how they relate to the Java implementation.
- A demonstration of the working code.
- Discussion on advantages and potential improvements.

Presentation dates will be announced after the submission deadline. Every student in the
group must actively participate in the presentation. Projects that are not presented will not
be evaluated, even if the report is submitted. Late submissions will not be accepted.
